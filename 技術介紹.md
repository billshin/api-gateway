# API Gateway 技術介紹

以下介紹每個節點使用計術簡介

### Traefik

**所有流量的管控專家**

1. 反向代理 
    - 文件自動化配置
    - docker自動掃描    
    - 設定後無須指令重新載入

2. Middleware 介入管控
    - 流量限制
    - 認證機制(Auth API)

3. Load Balancer
    - 對於反向代理目標,提供負載均衡
    - 健康檢查 , 輪詢API並標示健康狀態



Traefik 分三層：

1️⃣ entrypoint  「門（port）」
2️⃣ router 「規則（path / host）」
3️⃣ service 「真正 API」



### Prometheus

**觀察趨勢專用**

1. 自動收集 Metrics 資料(定時pull)
    - scrape_interval 可設定多久要監控(預設15s,也是常見的值)
2. 使用 **PromQL** 查詢資料


#### Metrics 介紹
Metrics 固定格式的時間序列資料 格式如下
```
metric_name{label_key="label_value",label_key2="label_value2"} value
```

範例如下
```
http_requests_total{service="auth-api",status="200"} 1520
http_requests_total{service="auth-api",status="500"} 84
http_request_duration_seconds_bucket{service="auth-api",le="0.1"} 320
http_request_duration_seconds_bucket{service="auth-api",le="0.5"} 1100
cpu_usage{container="auth-api"} 0.82
memory_usage_bytes{container="auth-api"} 268435456
```
這些資料能回答：
Auth API 總共處理了多少請求
500 錯誤是不是突然升高
延遲是否變慢
CPU、記憶體是不是過高

#### Metrics 時間概念
至於這些紀錄並未標示時間戳,是因為 Prometheus 會自動加上 scrape 的時間戳,所以不需要在 Metrics 中額外標示時間
所以如果 Prometheus 每 15 秒抓一次，實際儲存會像概念上這樣
```
10:00:00 memory_usage_bytes{container="auth-api"} 268435456
10:00:15 memory_usage_bytes{container="auth-api"} 270532608
10:00:30 memory_usage_bytes{container="auth-api"} 269221888
```

時間都是由prometheus提供, 就沒有不同機器而有時間差的問題
至於怎麼去使用這資料


#### Metrics 類型

 1. counter - 增長型資料,只能增加不能減少,適合用來統計總量或事件次數
 2. gauge - 狀態型資料,可以增加也可以減少,適合用來統計當前狀態或資源使用量
 3. histogram - 分佈型資料,適合用來統計延遲或大小的分佈情況,會自動產生多個 bucket 來統計不同範圍的數值
 4. summary - 摘要型資料,適合用來統計延遲或大小的摘要情況,會自動產生多個 quantile 來統計不同百分位數的數值

(以下範例的時間只是參考,實際上不會有時間顯示在 Metrics 中,而是由 Prometheus 自動加上時間戳)
**counter 範例**
```
10:00:00 http_requests_total{service="auth-api",status="200"} 1520
10:00:15 http_requests_total{service="auth-api",status="200"} 1538
10:00:30 http_requests_total{service="auth-api",status="200"} 1551
```
因為它是 counter，所以通常不直接看 1520 本身，而是看它的「增加速度」。
Server 不重啟,這數值就無限累加,但也無所謂,要的是變化趨勢

**gauge 範例**
```
10:00:00 cpu_usage{container="auth-api"} 0.82
10:00:15 cpu_usage{container="auth-api"} 0.85
10:00:30 cpu_usage{container="auth-api"} 0.80
```
純粹看各時間的數值,不需要看變化趨勢,因為它是 gauge,可以增加也可以減少

**histogram/summary 範例**
```
10:00:00 http_request_duration_seconds_bucket{service="auth-api",le="0.1"} 320
10:00:00 http_request_duration_seconds_bucket{service="auth-api",le="0.5"} 1100
10:00:00 http_request_duration_seconds_bucket{service="auth-api",le="+Inf"} 1520
```
可以知道各區間的請求數量,也可以算出平均值、百分位數等統計資料, 但依然是看時間內的變化趨勢,而不是細節

Metrics長處: 善於統計時間內變化, 而不是細節

#### 關於Metrics Label

先說結論, **請不要設定過多label!!**

Metrics Label 是一種 key-value 的標籤,用來描述 Metrics 的屬性或維度
例如:
```
http_requests_total{service="auth-api",status="200"} 1520
http_requests_total{service="user-api",status="500"} 200
```
而 http_requests_total 中label
service: auth-api, user-api
status: 200, 500 
實際上會變成 2 x 2 = 4 條 time series

也就是label不應該放入過多的維度,否則會產生過多的 time series,導致 Prometheus 的性能下降
例如 user_id 這種維度就不適合放在 label 中,這種Cross join 值接產生許多細碎的 time series
(但可以用 /api/action/:id 這種方式統計,但:id這寫法有寫等於沒寫)
實際上要關心的其實應該是 這時間是否API有問題 , 然後再去Loki查詢詳細資料
